Here are the definitive answers to your discovery questions. Proceed with the implementation based on these architectural boundaries.

### Branch 1: CRD Schema Design
*   **Q1.1 (Infrastructure sizing in CRD?): No.** Infrastructure sizing (CAPI `ClusterClass`, node counts, machine types) remains in the GitOps repository as separate CAPI manifests, managed by the CLI/GitOps. The `HubEnvironment` CRD is strictly for **in-cluster Day-2 configuration** (secrets, DB roles, messaging, identity). Do not mix infrastructure provisioning with app-layer orchestration.
*   **Q1.2 (JetStream schema?): Simplified schema.** Expose a simplified intent-based schema (e.g., just `name` and `subjects`). The Operator encapsulates the domain expertise and hardcodes the required defaults (retention limits, replicas) to prevent misconfiguration.
*   **Q1.3 (DB migrations?): Natively in the Operator.** Delete the Kubernetes Jobs. The Operator should use the `golang-migrate/migrate` library to execute schema migrations directly during the reconciliation loop.

### Branch 2: Operator Reconciliation Logic
*   **Q2.1 (Configurable passwords?): No.** Keep it fixed at 32-character secure random strings. YAGNI (You Aren't Gonna Need It). Optimization here adds unnecessary complexity to the CRD.
*   **Q2.2 (Database connection?): A (Read CNPG superuser secret).** The Operator acts as the platform administrator. Reading the `platform-db-superuser` secret generated by CNPG to execute `CREATE ROLE` is the standard, idiomatic approach for operators.
*   **Q2.3 (API retry strategy?): Fail and requeue.** Return `err` from the `Reconcile` loop. `controller-runtime` automatically handles exponential backoff. Do not write custom retry loops inside the Reconciler.

### Branch 3: Sync Wave Architecture
*   **Q3.1 (Application splitting?): Same Application.** Put the Operator Deployment and the `HubEnvironment` CR in the same ArgoCD Application, separated by Sync Waves (Wave 1 for Operator, Wave 2 for CR).
*   **Q3.2 (CRD packaging?): Packaged with the Operator.** Keep the CRDs in the standard `crds/` directory generated by Kubebuilder/Kustomize. 
*   **Q3.3 (OwnerReferences?): Yes.** The Operator must set `ownerReferences` on all Secrets, ConfigMaps, and Jobs it creates so they cascade-delete if the `HubEnvironment` CR is removed.

### Branch 4: External API Integration
*   **Q4.1 (Hydra SDK vs HTTP?): Use the official Go SDK.** (`github.com/ory/hydra-client-go`). It provides type safety and handles API versioning.
*   **Q4.2 (Hydra readiness?): Depend strictly on `/health/ready`.** Hydra's readiness probe inherently checks its own database migration status. If it returns 200 OK, it is safe to register clients.
*   **Q4.3 (NATS SDK vs CLI?): Use the NATS Go SDK.** Never `exec` CLI commands from an operator. The Go SDK is idiomatic, robust, and easily mockable for unit tests.

### Branch 5: State Management & Status
*   **Q5.1 (Condition types?): Both.** Use standard condition structures (`type`, `status`, `reason`, `message`), but use custom, granular Types (e.g., `Type: SecretZeroGenerated`, `Type: DatabaseConfigured`, alongside a top-level `Type: Ready`).
*   **Q5.2 (Status update frequency?): After EACH step.** This is a slow bootstrapping process. Updating the status after each step provides critical observability to the Platform Admin via `kubectl describe hubenvironment`.
*   **Q5.3 (Failure status?): `False` with error message.** If an API returns 500, set the condition Status to `False`, Reason to `ApiError`, and Message to the error string.

### Branch 6: Testing Strategy
*   **Q6.1 (E2E framework?): KUTTL.** Use KUTTL for declarative E2E testing against a real local cluster (Kind). Use `envtest` only for fast local unit/integration tests of the reconciler logic.
*   **Q6.2 (DB mocks vs Real?): Real database.** For KUTTL E2E tests, deploy an actual CNPG instance to verify the Go SQL drivers and network policies work correctly.
*   **Q6.3 (API mocks vs Real?): Real instances.** E2E tests must validate the actual helm charts and network boundaries. Deploy lightweight instances of Hydra and Infisical in the test cluster.

### Branch 7: CLI Boundary & Migration
*   **Q7.1 (Keep `--upgrade` in CLI?): Yes.** Keep `--upgrade` in the CLI, but **only** for CAPI infrastructure upgrades (e.g., `clusterctl upgrade`). Application-level upgrades (Ory, Infisical) are purely GitOps-driven.
*   **Q7.2 (Kubeconfig saving?): Stay in the CLI.** The CLI creates the initial cluster. The Operator doesn't exist yet to save the kubeconfig. The CLI must retrieve and save `~/.zero-ops/clusters/{name}.kubeconfig`.
*   **Q7.3 (Backward compatibility?): Break compatibility.** Moving from imperative to declarative is a major architectural shift (v9.0). Clean slate.

### Branch 8: Operator Deployment & Packaging
*   **Q8.1 (Packaging?): Kustomize.** Use the native `config/` directory structure generated by Kubebuilder. ArgoCD natively supports Kustomize.
*   **Q8.2 (Replicas?): HA mode.** Use the default leader election provided by Kubebuilder. It costs almost nothing and prevents split-brain scenarios if you ever scale up to 2 replicas for high availability.
*   **Q8.3 (RBAC scope?): Cluster-scoped.** The operator needs a `ClusterRole` to watch CNPG clusters, Namespaces, and potentially cross-namespace Secrets.
